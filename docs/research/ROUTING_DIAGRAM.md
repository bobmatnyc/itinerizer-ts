## Data Loading Strategy

### List View
1. Mount â†’ `loadItineraries()`
2. Fetch from API: `GET /api/v1/itineraries`
3. Render grid

### Detail View
1. Mount â†’ Read `$page.params.id`
2. `$effect` triggers `selectItinerary(id)`
3. Fetch from API: `GET /api/v1/itineraries/{id}`
4. Render detail + chat

### Caching
- Store maintains selected itinerary in memory
- Switching between itineraries fetches fresh data
- List is cached until explicit reload

## Browser Behavior

| Action | Old Behavior | New Behavior |
|--------|--------------|--------------|

| Refresh page | Lost selection | Preserves URL-based view |
| Back button | No effect | Returns to previous route |
| Share URL | Always home | Can share specific itinerary |
| Bookmark | Home page | Can bookmark itinerary |

## Performance Considerations

### Benefits
- **Code splitting**: List and detail views load separately
- **Lazy loading**: Detail view only loads when navigated to
- **SSR support**: Can pre-render itinerary details server-side
- **Smaller bundles**: Each route has minimal dependencies

### Trade-offs
- **Extra navigation**: List â†’ Detail requires route change (minimal overhead)
- **Data fetching**: Each navigation triggers new API call (can be optimized with caching)

**Framework:** SvelteKit 2 with file-based routing
